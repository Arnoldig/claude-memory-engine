## [Не выпущено]
### Исправлено
- `applies_to` одиночной СТРОКОЙ (`applies_to: "app/x.py"`) — валидный YAML-скаляр, который `applies_to._applies_globs` МОЛЧА ронял в `[]` с рождения модуля (в CHANGELOG отсутствует — не регресс, дыра изначальная). Механика: значение не начинается с `[` → код уходил в ветку разбора YAML-списка, первая же строка ниже (обычно `metadata:`) не начиналась с `- ` → `break` → пустой список. Ни ошибки, ни предупреждения — результат неотличим от «уроков нет», урок никогда не всплывал на правке файла. Разбор ветвится теперь по ЗНАЧЕНИЮ: `[` → инлайн-список; непустой скаляр → один глоб (кавычки снимает общий `strip_scalar`); ПУСТОЕ значение → YAML-список ниже (рабочий формат сохранён — скаляром считается только непустое значение). Значение в фигурных скобках (`{a: b}`, `{app,lib}/x.py`) глобом не считается: ни `fnmatch`, ни `glob` их не раскрывают → отдаётся пусто и срабатывает жалоба ниже с верным диагнозом (иначе такое значение уехало бы в отчёт ПРОТУХШИХ привязок как «путь не найден — файл переехал?», то есть с диагнозом мимо). +5 тестов (скаляр без кавычек/в кавычках/с звёздочкой; граница «пустое значение + список ниже»; контракт `applies_to_value`).
### Добавлено
- Движок ЖАЛУЕТСЯ ВСЛУХ, когда `applies_to:` задан, но глобов из него не вышло (`applies_to.unparsed_applies_to`) — раньше любое непонятое значение молча давало пустоту, неотличимую от «привязок нет» (третий случай этой болезни: ср. `topic` чужой таксономии, тихо выпадавший в «Без темы», и Stop-гейт, молчавший на русскую фразу закрытия). Два канала: (1) НЕМЕДЛЕННО в момент записи урока — PostToolUse `post-record` (`applies_to.unparsed_warning`) называет файл и само непонятое значение, пока автор дефекта рядом; (2) СВОДКА по всей памяти — `staleness.scan_unparsed` → секция в `_stale_pending.md`, которую следующий SessionStart печатает целиком и без троттлинга, пока не починят (соседняя полка к протухшим привязкам). В стража правки жалоба намеренно НЕ ставилась: сломанный урок не привязан к правимому файлу и сыпался бы на каждую правку любого файла. Ключ `applies_to_value` отличает «поля нет» (`None`) от «поле есть, значение пусто» (`""`) — на этом различии стоит жалоба. Тексты переопределяемы через `messages`. +7 тестов (обнаружение всех непонятых форм; обе жалобы; молчание на исправных уроках, на чтении и вне каталога памяти).

## [0.9.6] — 2026-07-13
### Добавлено
- Версия ЛОГИКИ парсера в отпечатке кэша ретривера (`memory_retrieve._PARSER_LOGIC_VERSION` → `_params_fingerprint`): раньше отпечаток покрывал только ПАРАМЕТРЫ токенизации (стем/мин.токен/окно тела/стоп-слова), но не саму логику разбора (`read_fields`/`tokenize`/`_parse_doc`). Свежесть строки SQLite-кэша проверяется по `mtime_ns`+`size` файла-урока — поэтому не-1:1 правка КОДА парсера при неизменных файлах и параметрах оставляла в кэше стемы, посчитанные СТАРЫМ парсером (тихо неверный поиск; для 0.9.4/0.9.5 приходилось вручную `rm _retrieve_cache.sqlite3*`). Теперь достаточно поднять `_PARSER_LOGIC_VERSION` на +1 при семантической правке разбора → отпечаток меняется → кэш обнуляется сам. +2 теста (смена версии меняет отпечаток; bump реально обнуляет строку кэша, а не отдаёт стемы старого парсера).
### Изменено
- Снятие обрамляющих кавычек у скалярных полей frontmatter вынесено в единый хелпер `applies_to.strip_scalar` и переиспользовано всеми парсерами (`catalog_generate.parse_frontmatter`, `memory_retrieve.read_fields`, `applies_to`, `staleness`) вместо копий идиомы `.strip().strip('"').strip("'")`. Чинит рассинхрон: `applies_to._DESC_RE` и `staleness._DESC_RE` снимали только пробелы → урок с `description: "…"` в кавычках показывался С кавычками в «уроках по пути» и `_stale_pending`, но БЕЗ — в CATALOG и поиске. Только отображение (на ранжирование и состав CATALOG не влияет). Поведение для незакавыченных значений 1:1 (255 прежних тестов зелёные); +3 теста (снятие кавычек в path-lessons/stale/archive + контракт хелпера).

## [0.9.0] — 2026-06-30
### Добавлено
- Страж актуальности линейки моделей (`llm_actuality`, новый модуль) — единый страж вместо двух прежних модельных проверок. Совмещает: (1) реактивно на `SessionStart` — если модель сессии не из подтверждённых семейств, сразу нудж «незнакомая модель»; (2) суточно — раз в `llm_actuality_interval_hours` (по умолч. 24) просит ассистента сверить линейку моделей (делегировав дешёвой модели + веб-поиск) и записать итог. Движок офлайн и сам в сеть не ходит. Итог пишется командами `cme_hook.sh llm-verified` (без изменений) / `llm-changes "<что изменилось>" --families <a,b,c>` в приватный файл состояния `_llm_registry_state.json` (он троттлит «раз в сутки» между сессиями и хранит подтверждённый список семейств — сид из `known_model_substrs`). Статус сверки выводится строкой в чек-листе закрытия («LLM actuality: verified <дата>» / «⚠ changes …» / «not verified yet»).
- Новые поля конфига: `llm_actuality_enabled` (по умолч. `true`) и `llm_actuality_interval_hours` (по умолч. `24`).
### Изменено
- На `SessionStart` подстраховку моделей теперь делает `llm_actuality` (реактивная проверка + суточная просьба) вместо прежнего таймера `model_registry_guard` по дате `model_registry_verified_on`. Модуль `model_registry_guard` сохранён (его `resolve_model`/`_is_known` и сообщение `model_registry.unknown_model` переиспользует `llm_actuality`); поля `model_registry_verified_on` / `model_registry_max_age_days` оставлены для обратной совместимости, но в дефолтном пути больше не используются.

## [0.8.0] — 2026-06-30
### Добавлено
- Страж устаревших уроков (`stale_reconcile`) теперь срабатывает на ФРАЗУ ЗАКРЫТИЯ СЕССИИ (хук `UserPromptSubmit`), а не на закрывающий коммит. Когда сообщение пользователя совпадает с новым `session_close_pattern`, в контекст подаётся ЧЕК-ЛИСТ итогов памяти: счётчики (показано уроков при правках / актуализировано / осталось), точные кандидаты на пере-проверку (правил файл с привязанным уроком, но урок не обновил), смысловой список связанных уроков (теперь считается ВСЕГДА при наличии правленых файлов — даже без `applies_to`-привязок), статус сроков годности/архива и список включённых/выключенных стражей. Чек-лист показывается ВСЕГДА при фразе закрытия, даже когда чисто (строка «протухших уроков нет»). Триггер не зависит от коммита/PR/скилла → работает и у проектов без команды закрытия. `/compact` не меняет `session_id` (метки видны), `/clear` страхуется бэкстопом `SessionEnd`.
- Новые поля конфига: `session_close_pattern` (regex фразы закрытия; пусто → чек-лист по фразе выключен) и `session_close_case_sensitive` (учитывать ли регистр; `true` убирает ложные срабатывания на частых словах вроде «Done»).
### Изменено
- Стражи включены «из коробки» (касается ВСЕХ потребителей): `stale_reconcile_gate` `False → True`, `archive_stale_months` `0 → 6`, `lesson_count_warn` `0 → 500` (подсказка про дубли на больших коллекциях) и `known_model_substrs` `() → (opus, sonnet, haiku, fable)` (реактивная проверка «незнакомая модель»; периодическая сверка линейки — отдельно, через `model_registry_verified_on`, по умолчанию выкл). Чек-лист всегда честно показывает, что включено и что найдено, поэтому включение не создаёт ложного впечатления работы. Без `applies_to`-привязок точный список пуст, но смысловой список и сводка стражей всё равно полезны. Прежнее поведение возвращается установкой этих полей в `false`/`0`/`()`.
### Удалено
- Прежний триггер `reconcile_reminder` (разовый блок на закрытии задачи в событии `Stop`) убран — его заменил чек-лист на фразу закрытия. Отдельный привратник «коммит закрыл задачу, а урока про неё нет» (`closure_reminder` / `task_close_lesson_gate`) НЕ затронут.

## [0.7.3] — 2026-06-29
### Исправлено
- В дефолтном `task_close_pattern` граница слева заменена с `\b` на негативный lookbehind `(?<![\w-])`. `\b` срабатывает и ПОСЛЕ дефиса, поэтому сообщение коммита вида `prefixed-closes #10` (или `auto-closes #10`) ложно распознавалось как закрытие задачи #10 — `extract_closed_task` возвращал `10` вместо `None`, и привратники закрытия (`closure_reminder`, `stale_reconcile`) могли показать лишнее одноразовое напоминание. Lookbehind по-прежнему пропускает обычные `Closes #id` / `Fixes #id` (в начале строки, после пробела или двоеточия), но не `<слово>-closes #id`. Найдено red-team-проверкой при добавлении русской формы закрытия. Затрагивает только generic-дефолт; потребителям, чей собственный `task_close_pattern` использует `\b` как левую границу перед словом закрытия (в т.ч. если прежний дефолт был скопирован в конфиг), стоит применить ту же замену `\b` → `(?<![\w-])`.

## [0.7.2] — 2026-06-28
### Исправлено
- `extract_closed_task` теперь возвращает ПЕРВУЮ непустую capture-группу совпадения, а не жёстко группу 1, — это позволяет проектному шаблону `task_close_pattern` распознавать закрытие в РАЗНЫХ формах записи, где id стоит по разные стороны ключевого слова. Англ. «Closes #id» кладёт id ПОСЛЕ слова, а локализованная форма вроде русской «#id закрыт» — ДО, и одной capture-группой обе не покрыть. Библиотечный дефолт остаётся английским (generic) — нужную форму проект задаёт в своём `task_close_pattern`. Заодно шаблон без групп больше не падает (None вместо IndexError). Найдено на реальном пропуске закрытия задачи русской фразой «#id закрыт» (без `Closes`): и `closure_reminder`, и `stale_reconcile` молча не сработали. Второй фикс распознавания закрытия после `%B`-фикса v0.7.1 (тело коммита).

При коммите-закрытии (`Closes #N`) Stop ОДИН раз показывает уроки, привязанные к файлам, которые правились в этой сессии, но сами не были актуализированы, — с вопросом «не устарели ли?» (поправить / пометить неактуальным / заменить новым).
