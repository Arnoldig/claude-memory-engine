## [0.10.0] — 2026-07-17
### Исправлено
- **Страж перестал слепнуть на уроках, которые сам же велел записать.** Движок определял урок по началу имени файла (`feedback_`/`reference_`/`project_`), но файлы уроков создаёт НЕ он: их пишет встроенная авто-память Claude Code (её подпись `metadata.originSessionId` стоит в каждом файле; доказано на репозитории самого движка, где библиотека не установлена, а 10 файлов уроков есть). Урок, названный иначе, для каталога и ретривера существовал, а для стража — нет: страж требовал зафиксировать урок, урок записывали, страж требовал снова. Читалось как «хук сломан и ругается всегда». Теперь уроком считается ЛЮБОЙ `*.md` в корне `memory_dir`, кроме ядра, указателя и приватных `_*` — единый источник истины `lesson_files.is_lesson_file`/`lesson_paths`, общий для стража, привратника закрытия, каталога, ретривера, `applies_to`, `staleness` и bloat-check. **Миграция не нужна: ни один урок не требует переименования или переноса.** Расширение строго МЯГЧЕ — оно может только снять блок, но не создать новый. Замер на боевых корпусах: в поле зрения стража возвращается +5 уроков (427) и +13 (87).
- **Тип урока берётся из поля `type`, а не угадывается по имени файла** (`lesson_files.lesson_type`). Модель именует файл по формуле `<значение type>_<слаг>.md` (замер: у 421 файла из 426 приставка ТОЧНО равна полю), но типов в ходу девять, а `lesson_prefixes` знал три. Поле `type` присутствует в 513 файлах из 514 — оно надёжно, имя нет. Фолбэка на приставку намеренно НЕТ: он законсервировал бы то самое второе определение урока. Файл без поля получает пустой тип — честный результат, видимый в пульсе здоровья.
- **`user` добавлен в дефолтные `lesson_prefixes`.** Официальный словарь типов авто-памяти Claude Code — `user | feedback | project | reference`, а дефолт библиотеки знал три из четырёх: урок типа `user` был невидим стражу в ЛЮБОМ проекте на дефолтах, по конструкции (в обоих боевых корпусах — ровно по одному такому).
- **Установщик больше не разводит каталоги по умолчанию.** `claude-memory init` без `--memory-dir` прописывал `~/.claude/memory` — каталог, в который не пишет никто, тогда как Claude Code держит память в `~/.claude/projects/<slug>/memory`. То есть разъезд каталогов был ПОВЕДЕНИЕМ ИЗ КОРОБКИ, а не ошибкой пользователя: движок читал пустую папку, страж требовал урок, которого там никогда не появится. Теперь путь ищется у хозяина (`claude_code_env.resolve_auto_memory_dir`): подтверждённый диском — берётся молча, невыведенный — берётся с явной оговоркой и БЕЗ создания каталога (пустышка рядом с настоящей папкой замаскировала бы ошибку от самодиагностики). Из git-worktree корректно находится каталог основного чекаута — память общая на весь репозиторий.
- **Сверка `[[ссылок]]` между уроками больше не пропускает уроки без приставки** (`catalog_generate.find_broken_wikilinks`). Валидировались только цели, начинающиеся с префикса урока: в боевом корпусе ВСЕ связи kebab-уроков не проверялись вообще — оборвись они, никто бы не узнал. Признак ссылки теперь ФОРМА цели (составной слаг, опц. `.md`), а не имя файла. На боевых корпусах вскрылось 10 и 1 настоящих оборванных ссылок.
- **Привратник закрытия задачи узнаёт ВСЕ девять слов-закрытий GitHub.** Дефолтный `task_close_pattern` знал `close/closes/closed` и `fix/fixes/fixed`, но не семью `resolve/resolves/resolved` — то есть треть законных форм закрытия GitHub привратник молча не узнавал, а «не узнал» неотличимо от «задачу не закрывали». Поймано при переводе трекера с файлов `.md` на GitHub Issues. Тот же класс дефекта уже описан в памяти проекта (страж, узнающий событие по НЕПОЛНОМУ списку форм, недосрабатывает молча): список форм обязан зеркалить ИСТОЧНИК — документацию GitHub, — а не интуицию автора. +3 теста, включая параметризованный по всем девяти словам и контроль «голое упоминание `(#5)` — не закрытие».
- **`staleness` и `applies_to` переведены на общий набор уроков.** Они глобили `*.md` вообще без исключения ядра/указателя/приватных — третий вариант определения урока, живший своей жизнью. Сегодня не стреляло (у ядра нет frontmatter), но это ровно та мина, что подорвалась у стража.
### Добавлено
- **Самодиагностика расхождений с настройками хозяина** (`self_check.settings_issues`, SessionStart + CLI). Три жалобы, от смертельной к бытовой: (1) авто-память Claude Code ВЫКЛЮЧЕНА (`autoMemoryEnabled: false` или env `CLAUDE_CODE_DISABLE_AUTO_MEMORY`), а стражи уроков включены → уроки писать НЕКОМУ вовсе (движок их не создаёт, он читатель), Stop блокирует после каждого коммита и удовлетворить его нечем — единственная комбинация с настоящим вечным тупиком, достижимая одной строкой в settings.json; (2) `autoMemoryDirectory` задан явно и ≠ `memory_dir`; (3) `memory_dir` пуст, а в папке авто-памяти уроки есть — наследие сломанного дефолта установщика. Жалоба (3) выдаётся ТОЛЬКО когда догадка подтверждена диском (своих уроков ноль И у хозяина они есть): правило вывода слага не задокументировано, и гадать вслух движок не вправе. Fail-open: нет файла / битый JSON / git недоступен → молчание.
- **Отчёт самопроверки**: `python3 -m claude_memory.self_check` печатает КАРТИНУ всегда, а не только жалобы — путь движка, путь авто-памяти Claude Code, совпадают ли они, сколько уроков видно и каких типов. Жалобы отвечают «что сломано», и при чистом конфиге их нет — человеку было нечем ПРОВЕРИТЬ настройку, только узнать о поломке. А проверять надо: разъезд каталогов выглядит ровно как «всё хорошо, уроков просто пока нет».
- Новый модуль `claude_code_env` — знание о среде-хозяине: правило слага (выведено обратной инженерией, проверено на трёх боевых проектах включая кириллические пути), каталог основного чекаута из worktree, области `settings.json` (local > project > user), `autoMemoryDirectory`, киллсвитчи авто-памяти. Догадка никогда не подаётся как истина: либо подтверждена диском, либо сопровождается оговоркой.
- Поле конфига `precedent_files` — явные файлы-накопители прецедентов для авто-архивации. Контракт «`lesson_prefixes[0]` — это файл прецедентов» был позиционным и нигде не описанным: проект с другим порядком префиксов молча целился не в тот файл (живой случай — первый префикс `urok`). `None` → историческое поведение бит-в-бит.
- +70 тестов, включая анти-воскрешение: `test_lesson_files.test_all_consumers_see_the_same_set` требует, чтобы каталог, ретривер и страж физически видели ОДИН набор. Пока тест зелёный, баг не воскреснет. Плюс сквозной сценарий «свежий коммит → блок → записали kebab-урок → блок снят», которого раньше не было (иначе баг бы не дожил).
### Изменено
- `lesson_prefixes` **УСТАРЕЛО** и больше не отвечает на вопрос «что такое урок» — на него отвечает `lesson_files.is_lesson_file`, а тип берётся из `type`. Поле СОХРАНЕНО (конфиги внешних потребителей не ломаем) в узких ролях: фолбэк для `size_warn_prefixes` и `precedent_files`, ссылки в `precedent_index`.
- `size_warn_prefixes: None` теперь значит «все уроки», а не «уроки с приставкой»: урок без приставки рос без единого предупреждения. Само поле остаётся СУЖАЮЩЕЙ ручкой.
- Граница `self_check` переформулирована ПО ФУНКЦИЯМ, а не по модулю: право бежать на каждом SessionStart принадлежит `warnings()`/`run()` (конфиги + метаданные каталогов, содержимое уроков — никогда), а `report()` живёт только в verbose-CLI по явной просьбе человека и содержимое читает (разбивка по типам ради того и заведена). Модульный инвариант «никогда не читаем уроки» опровергался бы собственным `report()` — и следующий человек «починил» бы противоречие, либо ослабив инвариант до бессмыслицы, либо выпотрошив отчёт.
- `settings_issues` на здоровом проекте ускорен 15.3 мс → 0.7 мс: дешёвый glob «вижу ли я уроки» стоит ПЕРЕД git-вызовом (14 мс), поэтому за диагностику платит только тот, кто уже сломан. Чтение settings.json — 0.15 мс на область, в оптимизации не нуждается.
- Удалён `catalog_generate._skip_basenames` — мёртвая функция и уцелевший обломок второго определения урока: рабочая, вызываемая одной строкой, с виду легитимная. Ровно так болезнь и возвращается — в релизе, чья суть «определение одно», готовая копия старого не имеет права жить.

## [0.9.7] — 2026-07-16
### Добавлено
- Предупреждение о раздутом `description` в момент записи урока (`bloat.description_long`, PostToolUse `bloat-check`, порог `description_warn_chars`, по умолч. 500 знаков, 0 = выкл). `description` — это КРАТКОЕ содержание, и страж правки с ретривером печатают его ЦЕЛИКОМ: несколько раздутых уроков на одном файле дают стену текста, которую перестают читать (тогда привязки бессмысленны). Обрезать при показе — не лечение: теряется контекст, ради которого урок и писали, поэтому чиним источник и в момент, когда автор рядом. Порог — ~2.5× медианы боевых корпусов (202 и 173 знака), то есть задевает не «подробные» уроки, а те, где тело переехало в описание или склеены две темы (на корпусах: 5 из 427 и 12 из 87; живой пример — описание на 1209 знаков, вторая половина которого начинается с «CSS issue #3», то есть в файле два разных урока). Срабатывает только на записи — существующие уроки не нудят. +3 теста.
- Движок ЖАЛУЕТСЯ ВСЛУХ, когда `applies_to:` задан, но глобов из него не вышло (`applies_to.unparsed_applies_to`) — раньше любое непонятое значение молча давало пустоту, неотличимую от «привязок нет» (третий случай этой болезни: ср. `topic` чужой таксономии, тихо выпадавший в «Без темы», и Stop-гейт, молчавший на русскую фразу закрытия). Два канала: (1) НЕМЕДЛЕННО в момент записи урока — PostToolUse `post-record` (`applies_to.unparsed_warning`) называет файл и само непонятое значение, пока автор дефекта рядом; (2) СВОДКА по всей памяти — `staleness.scan_unparsed` → секция в `_stale_pending.md`, которую следующий SessionStart печатает целиком и без троттлинга, пока не починят (соседняя полка к протухшим привязкам). В стража правки жалоба намеренно НЕ ставилась: сломанный урок не привязан к правимому файлу и сыпался бы на каждую правку любого файла. Ключ `applies_to_value` отличает «поля нет» (`None`) от «поле есть, значение пусто» (`""`) — на этом различии стоит жалоба. Тексты переопределяемы через `messages`. +7 тестов (обнаружение всех непонятых форм; обе жалобы; молчание на исправных уроках, на чтении и вне каталога памяти).
### Изменено
- Списочные поля конфига приводятся к каноническому виду в `MemoryConfig.__post_init__`, если описка ОДНОЗНАЧНА по намерению: `retrieve_extensions: [".py"]` → `py`, `watched_dirs: ["./app/"]` → `app`, `lesson_prefixes: ["feedback_"]` → `feedback`, `staleness_skip_dirs: [".git/"]` → `.git`. Здесь движок намеренно НЕ жалуется, а понимает — и это не исключение из правила «не отменять молча явно заданное», а следование ему: настройка начинает РАБОТАТЬ вместо того, чтобы тихо выпадать. Жалоба остаётся там, где намерение неясно (описка в ИМЕНИ ключа → `self_check.typo_key_issues` спрашивает), а `.py` не может означать ничего, кроме `py`. Каждый случай до этого ломался молча и воспроизведён: `.py` → `_path_re` собирал `\.(?:\.py)` (требовал `..py`) → канал «уроки по пути из запроса» не находил ничего; `app/` → там же требовалось `app//`; `feedback_` → `catalog_generate` клеил `feedback__`, `stop_check` искал `feedback__*.md` → движок считал, что уроков нет вообще, и Stop-страж переставал видеть записанное (при этом `startswith` в другом месте работал — поведение расходилось между частями движка); `.git/` → сверка идёт с голым именем каталога от `os.walk` → пропуск не срабатывал и обход лез в тяжёлые каталоги. Нормализация в `__post_init__`, а не в `_coerce`: инвариант обязан держаться при ЛЮБОМ способе создания конфига (JSON / дефолты / `dataclasses.replace`), иначе разойдутся боевой путь и тесты. Элементы, схлопывающиеся в пустоту (`"."`, `"/"`, `"_"`), выбрасываются — пустая ветка в regex совпадала бы с чем угодно. На живых конфигах обоих проектов изменений 0 (там всё каноническое). +4 теста.
- Служебный каталог `.claude/` больше не пропускается стражем правки СКОПОМ — правило стало точным и переехало из `hooks_cli` в `applies_to.find_lessons_for_path` (одно на оба канала): внутри `.claude/` матчат только те глобы, что сами адресуют `.claude/`. Прежний общий пропуск защищал от реального (широкий `*.py` — `fnmatch` `*` матчит и слэши — не должен всплывать на служебных файлах), но заодно молча отменял ЯВНЫЕ привязки к настоящему коду проекта, который там живёт: у одного проекта в `.claude/hooks/` лежат стражи выкладки, у другого есть отдельный урок, ПРЕДПИСЫВАЮЩИЙ привязывать уроки о движке к вендоренной копии (`.claude/memory_engine/claude_memory/…`) — то есть движок глушил собственное задокументированное соглашение проекта. Итого 12 уроков в двух проектах не всплывали никогда. Вдобавок половины движка расходились: `memory_retrieve._path_re` `.claude/` НИКОГДА не пропускал, и по упоминанию пути в запросе урок находился, а на правке того же файла молчал. Теперь обе правды уживаются. Проверено на живых проектах: срабатывает ровно на явно привязанных файлах (8 из 72 и 6 из 33 файлов в `.claude/`), случайных срабатываний от широких глобов — 0; прежний регресс-замок (`test_pre_edit_gate_fires_in_worktree_but_skips_tooling`) зелёный без правок. +3 теста.
### Исправлено
- Поля-даты урока (`reverify_after`, `archived_on`) читались строгим regex `(\d{4}-\d{2}-\d{2})`, который СЛИВАЛ «поля нет» и «поле есть, но не дата». `reverify_after: "01.01.2026"` (естественная русская запись) МОЛЧА не существовал: урок выглядел time-bound и жил вечно; `archived_on` в том же виде навсегда проходил мимо `archive_stale_months`. Жалобы не было никакой — ни в пульсе, ни в `_stale_pending`. Теперь `staleness._date_or_complaint` даёт РОВНО три состояния (поля нет / разобрано / задано мусором), причём детектор и парсер — один код (разъехаться не могут, иначе получили бы либо ложную тишину, либо ложную тревогу). Общие хелперы `applies_to.iso_date_or_none` + `applies_to.field_value` — вместо трёх копий regex+try/except. Доказано 1:1 на боевых корпусах (19 полей с датами: 0 расхождений old↔new, 0 ложных жалоб); +6 тестов.
- Молча выключавшиеся стражи и опечатки в конфиге — теперь их ловит `self_check` (он уже бежит на КАЖДОМ SessionStart и задуман ровно как «поймать настройку, которая тихо портит сеанс», но проверял единственную вещь): (а) `self_check.typo_key_issues` — ОПЕЧАТКА в имени ключа (`session_close_patterns` с лишней `s` → настройка молча выброшена, остаётся английский дефолт; реинкарнация уже случившегося бага со стражем закрытия). Отбрасывание неизвестных ключей сохранено (forward-compat: старый движок × новый конфиг), но популяции разведены по близости к известному имени (`difflib`, stdlib): опечатка по построению близка, честно новый ключ — нет; ключи с ведущим `_` пропускаются всегда (принятое соглашение для заметок в JSON — у живого конфига есть `_task_close_pattern_note`, близкий к `task_close_pattern`, без правила жалоба была бы ложной каждый старт). Текст жалобы сформулирован ВОПРОСОМ («опечатка или ключ новой версии?») — ложное срабатывание не врёт, а спрашивает. (б) `self_check.bad_regex_issues` — битый пользовательский regex (`session_close_pattern`/`task_close_pattern`): движок ловил `re.error` и возвращал «не совпало» → страж молча выключался, а «выключен» неотличимо от «ничего не нашёл». (в) `self_check.bad_date_issues` — `model_registry_verified_on` не в ISO → страж сверки линейки молча выключался, хотя человек его ЗАПОЛНИЛ, то есть намеренно включал. Чтобы это стало возможным, `config._coerce` теперь ПОМНИТ выброшенные ключи в служебном `unknown_config_keys` (из JSON не принимается — иначе конфиг подделал бы отчёт о собственных опечатках): до этого self_check физически не мог увидеть опечатку, конфиг приходил к нему уже очищенным. Ноль ложных срабатываний на обоих живых конфигах; +7 тестов.
- `applies_to` одиночной СТРОКОЙ (`applies_to: "app/x.py"`) — валидный YAML-скаляр, который `applies_to._applies_globs` МОЛЧА ронял в `[]` с рождения модуля (в CHANGELOG отсутствует — не регресс, дыра изначальная). Разбор ветвится теперь по ЗНАЧЕНИЮ: `[` → инлайн-список; непустой скаляр → один глоб (кавычки снимает общий `strip_scalar`); ПУСТОЕ значение → YAML-список ниже (рабочий формат сохранён — скаляром считается только непустое значение). Значение в фигурных скобках (`{a: b}`, `{app,lib}/x.py`) глобом не считается: ни `fnmatch`, ни `glob` их не раскрывают → отдаётся пусто и срабатывает жалоба ниже с верным диагнозом (иначе такое значение уехало бы в отчёт ПРОТУХШИХ привязок как «путь не найден — файл переехал?», то есть с диагнозом мимо).
