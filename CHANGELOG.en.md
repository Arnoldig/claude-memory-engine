## [0.12.0] — 2026-07-18

### Fixed
- **Self-check no longer stays silent about diverged directories just because the engine "sees something"** ([#5](https://github.com/Arnoldig/claude-memory-engine/issues/5)). When `memory_dir` does not match the directory Claude Code writes its auto-memory to and `autoMemoryDirectory` is not set explicitly, `settings_issues()` complained ONLY if `memory_dir` was empty. One lesson in it was enough to silence the check — while `report()`, in that very same state, printed `same directory: NO`. New complaint: `self_check.memory_dir_divergent`.

  **Why the old silence was wrong.** It rested on "the user may keep a deliberate separate corpus". The module itself refutes that: lessons are created ONLY by Claude Code's auto-memory — the engine never writes them. So "directories diverged + lesson gates on" is unsatisfiable by construction: the gate demands a fresh lesson, the lesson lands in the other directory, and the engine will never see it. That is precisely the deadlock this check exists for, and people arrive there on their own: renaming or moving the repository directory changes the slug, Claude Code starts writing next door, and the previous corpus becomes a dead tail — which then silences the complaint by merely existing.

  **Scope of the complaint.** Silent when the gates are off (`stop_lessons_enabled`, `task_close_lesson_gate`) — a separately curated corpus is coherent there, with the engine acting as a plain retriever over it. Silent when auto-memory is disabled: "new lessons go elsewhere" would be a lie — they go nowhere, and a separate complaint already says so. As before, it requires confirmation from disk: the auto-memory directory must exist and be non-empty.

  **Invariant pinned by a test:** with gates on AND the divergence CONFIRMED on disk (the auto-memory directory exists and is non-empty), the state "`report()` prints `NO` while `settings_issues()` is empty" is now unreachable. The confirmation clause matters: if the auto-memory directory does not exist yet, `report()` still prints `NO` marked "derived, not confirmed" and no complaint is raised — deliberately, since otherwise the engine would be guessing out loud from an undocumented slug rule.

### Changed
- **The cost of the check is held for a main checkout.** The cheap gate that kept git out of the hot path was the "do I see any lessons" glob — but the new complaint is exactly about the case where lessons ARE visible. Replaced with one that is equally cheap and actually precise: for a MAIN checkout (`.git` is a directory and contains `HEAD`) the auto-memory path is derived without git at all, in two stats (`claude_code_env.default_auto_memory_dir_without_git`), and a match against `memory_dir` settles the question. The contract is pinned by a test that fails if the gate is removed.

  **Who does start paying:** a healthy project opened from a git worktree, or living outside git — there the `.git` shape proves nothing, so SessionStart gains one git call (~14 ms) that 0.12.0 did not have before. For a main checkout, nothing changes. A deliberate trade: before 0.12.0 that "free" path was bought by not performing the check at all.
- `claude_code_env.main_checkout` is now cached per process (`functools.lru_cache`): several call sites ask for it within one hook session, and the checkout root does not change inside a short-lived process.

## [0.11.0] — 2026-07-17

### ⚠ BREAKING CHANGE
- **`precedent_keyword` / `precedent_pointer` defaults are now English**: `"Прецедент"` / `"перенесён в"` → `"Precedent"` / `"moved to"`.

  **Who is affected:** only those who never set these fields in their config AND write precedent cards in Russian. **What to do:** add two lines to `.claude/claude-memory.config.json` —
  ```json
  "precedent_keyword": "Прецедент",
  "precedent_pointer": "перенесён в",
  ```
  No data is touched, no lesson is renamed. The engine also **says this out loud**: if a file still holds cards with the old Russian keyword while `precedent_keyword` sits at the library default, `bloat-check` warns on the next edit of that file with the exact fix (`bloat.precedent_legacy_keyword`) — a breaking change nobody notices is the very class this release is against.

  **Why it had to happen.** `memory_archive._precedent_re` is built from `re.escape(precedent_keyword)`. A Russian default meant that for EVERYONE writing cards in English who never touched the config, precedent auto-archiving never fired ONCE and `precedent_count_warn` counted NOTHING — silently, with the mechanism switched on. This is a "default that never works" — the same class as `~/.claude/memory` in 0.10.0 (a directory nobody writes to). The project's own rule ("a generic library default is language-neutral; language specifics belong in the project config") was written down in memory but checked by nothing — it is now pinned by `test_recogniser_defaults_are_language_neutral`. The Russian forms moved to `examples/claude-memory.config.ru.json`, with localisation pinned by its own test.

  How deep it sat: NINE of the engine's own tests silently lived on the Russian default — meaning the English path (what you get out of the box) was tested by nothing at all.

### Fixed
- **The Russian example config had fallen six releases behind.** `examples/claude-memory.config.ru.json` translated 76 of 133 messages: untranslated were the whole self-check (17), everything about staleness (11), archive pruning (6), diagnostics (5), the "could not parse this field" complaints (3) — precisely what 0.9.3→0.10.1 were built for. Nothing crashed: `msg()` falls back to the English default, so the "localised example" silently produced output in two languages at once. The English example was edited eight times running; the Russian one had not moved since 0.9.2, and no test ever looked inside the `messages` block. All 133 are now translated and pinned by a test.
- **Machine tags and a placeholder were lost in existing translations.** Eight translations had dropped their `[memory]`-style channel tag (a machine label, not prose), and one had a hard-coded CLI command instead of the `{extract_cmd}` placeholder — it would have gone stale on the first rename with nobody noticing.
- **A dead key in the message catalogue.** `stale_reconcile.reminder` survived the mechanism's replacement in 0.8.0 and hung around unused. Key names are a PUBLIC contract (consumers translate them by name), so a dead key turns someone's translation into a silent no-op: `self_check.message_placeholder_issues` skips orphan keys by design. The `test_every_msg_key_in_code_exists` lock was ONE-WAY (code → defaults); the reverse lock is added. Note for consumers: if you translated this key, your override becomes an orphan — harmless, but it does nothing (the key has been dead since 0.8.0).
- **The copy of the close rule in tests had drifted from the code.** `tests/conftest.py::RU_EN_CLOSE_PATTERN` knew six closing keywords instead of nine: 0.10.0 taught the default the `resolve` family, the copy was not updated — so part of the suite ran against a rule that does not exist in production. It is now compared by BEHAVIOUR (the copy is localised and must differ as a string, but not in the set of forms it recognises).
- **The README module table did not know about `lesson_files`, `claude_code_env` or `model_registry_guard`** — the README quietly lied about what the engine is made of. Caught by the owner by eye; pinned by "every package module is mentioned in the README" (both languages).

### Added
- **CI runs the tests on EVERY push and pull request** (`.github/workflows/tests.yml`). Previously the suite ran ONLY on release (`publish.yml`, `on: release published`): a pull request merged having verified nothing, and breakage surfaced at release time — when rolling back costs the most. It looked fine because there were no red checks, not because there were green ones.
- **Python matrix: 3.9 and latest.** `pyproject.toml` promises `>=3.9`, while CI ran `3.x` — so the compatibility promise was checked by nobody: 3.10+ syntax would pass green and break on the user's machine at install time. The invariant "CI runs the oldest supported version" is pinned by a test.
- `tests/test_repo_invariants.py` — a new file for one class of trap: a value must match in TWO places, nothing checks it, and the divergence is silent. It covers: version in `pyproject.toml` vs `__init__.py`; the `tests-NNN` badge vs the real count (during 0.10.0→0.10.1 that badge was hand-edited four times and went stale each time); the README module table; the close-rule copy in conftest; the Python matrix in CI. The rule now written into the project's memory: **a trap that has sprung twice must become a check, not another paragraph in a note.**
- +10 tests (403 vs 393).

## [0.10.1] — 2026-07-17
### Fixed
- **The recovery hint suggested a command that does not exist.** When the lessons path could not be confirmed, `claude-memory init` printed "re-run: `claude-memory init --memory-dir <correct path>`". There is no `--memory-dir` flag — the path is positional, and argparse answered `unrecognized arguments`. So the person whose path had already failed to resolve was handed a command that will not run: the message looked like help without being help — the same class of defect as the whole 0.10.0 release. It now prints a runnable form (`claude-memory init <project> <path>`), pinned by a test that parses the suggested command with the SAME parser — plausibility does not count (`test_cli.test_derived_path_warning_suggests_a_command_that_actually_runs`, red on 0.10.0).
- **Size / empty-`name` nudges fired on files in memory SUBDIRECTORIES — a 0.10.0 regression.** `hooks_cli` called `is_lesson_file` on a bare basename of an arbitrary path, but the name is only half the criterion: `lesson_paths` walks the ROOT of `memory_dir` only. The module created as the single source of truth was therefore giving TWO different answers to "what is a lesson" — the exact divergence it exists to eliminate. The irony: writing to `drafts/<session>.md` is demanded by the engine itself (`session_marker_guard`), a session breakdown is LONG by design, and its name is a random uuid you cannot list in `size_exempt`; a live project has that subfolder. Added `lesson_files.is_lesson_path` (root + name); whoever holds a path calls it. Pinned by two tests, including the invariant "`is_lesson_path` and `lesson_paths` never disagree".
- **The suite depended on the home directory of whoever ran it.** Since 0.10.0 the engine reads the host's settings, whose weakest scope is `~/.claude/settings.json`. The "we stay silent on a broken setting" tests passed only because the author has no `autoMemoryDirectory` there; with a `HOME` where it is set (a legitimate setting!), FIVE tests failed — THREE of them pre-existing, untouched by this work. A test that depends on someone's laptop pins the laptop, not the behaviour. A shared autouse fixture `isolated_home` (conftest) now swaps the home directory for EVERY test, on both channels (`Path.home` and env `HOME`). Verified: 390 green across four different `HOME`s, including auto-memory disabled and no `~/.claude` at all.
- **The flagship anti-resurrection test checked ONE NUMBER for the gate instead of the set.** Sets were compared for the catalog and the retriever, but the gate got only `max(mtime)`. Any narrowing that missed the newest file passed green — a gate silently losing `role`/`prompt`/`principle` types again, for instance, went unnoticed. The set is now proven ELEMENT BY ELEMENT (each file made newest in turn), plus the converse: core/catalog/private files made newest must NOT be seen by the gate.
- **The auto-memory path shape was pinned by nothing** — tests took the expected value from the function under test itself. Replacing `projects` with anything would restore the "engine looks for lessons in a nonexistent folder" defect with the suite still green. The `~/.claude/projects/<slug>/memory` shape is the central fact of the release and has no higher contract (the slug rule is undocumented), so it is now pinned by literal path segments.
- **Stale CLI and `install.sh` texts.** The `install.sh` header promised "creates the memory directory if it does not exist yet" — on a derived path the directory is deliberately NOT created (an empty twin next to the real folder would mask the self-check's own complaint). The `doctor` help read "config self-check (message placeholders)" — there are five checks, including the important one: comparing paths against Claude Code's settings.
- **`install.sh --help` was silently truncated.** It is assembled from the header comment, but the range was hardcoded by LINE NUMBERS (`sed -n '2,16p'`): any header edit dropped the tail, and the whole Usage section stopped printing. `--help` still printed *something* — a truncated help is indistinguishable from a complete one unless you know what should follow. Caught by dogfooding: editing the header in this very release, I shifted the lines and truncated Usage myself. It now prints the entire leading `#` block (recognising "a comment line" needs no sync with the header's length), pinned by `tests/test_install_sh_help.py`, which compares the output against the header LINES from the file rather than a counter.
- **The example config had drifted from the code and handed back an already-fixed bug.** `examples/claude-memory.config.json` carried the OLD `task_close_pattern` — six closing keywords instead of nine. People copy the example when setting the engine up, so you could install the fresh 0.10.0 engine, copy the example, and get back exactly the failure that release fixed: `Resolves #12` goes silently unrecognised and the task-close gate never fires. "Keep examples in sync with the defaults" was already a known release-process trap — known to a human, unchecked by a machine, and therefore reproducible precisely when a release is in a hurry. Now a machine checks it: `tests/test_examples_sync.py` compares the example's neutral defaults against the code BY VALUE and separately BY BEHAVIOUR (the example must recognise all nine closing forms — a deliberate edit could otherwise drift while still looking "close enough"), and it also catches unknown keys in the example: a typo in a key name would propagate to everyone who copied it.
- **The README (both languages) recommended a configuration that BREAKS the engine**: "the same memory path for all projects gives a shared pool of lessons". The engine itself cannot give you a shared pool — lessons are written by Claude Code's auto-memory, which keeps memory per project by default; `memory_dir` pointed at a shared folder means reading a directory nobody writes to. (Only Claude Code can put every project in one folder, via `autoMemoryDirectory` — now documented.) Also: the quick start promised that `init` "creates the lessons folder" (it LOCATES it and deliberately does NOT create it on a derived path); the self-check was described as one check out of five — omitting the one that compares paths against Claude Code; `doctor` was named as the verification command, while only `python3 -m claude_memory.self_check` prints the setup picture; and "the engine only reads lessons" was false — it does write into existing files (auto-archiving cuts out old precedent cards). Tests badge: 200+ → 390.

## [0.10.0] — 2026-07-17
### Fixed
- **The Stop gate no longer goes blind to the very lessons it demands.** The engine identified a lesson by its filename prefix (`feedback_`/`reference_`/`project_`), but it does not create lesson files — Claude Code's built-in auto-memory does (its `metadata.originSessionId` signature is in every file; proven on the engine's own repository, where the library is not installed yet 10 lesson files exist). A lesson named otherwise existed for the catalog and the retriever but not for the gate: the gate demanded a lesson, the lesson got written, the gate demanded again. It read as "the hook is broken and always complains". A lesson is now ANY `*.md` in the root of `memory_dir` except the core file, the catalog and private `_*` files — one source of truth (`lesson_files.is_lesson_file`/`lesson_paths`) shared by the Stop gate, the task-close gate, the catalog, the retriever, `applies_to`, `staleness` and bloat-check. **No migration: not a single lesson needs renaming or moving.** Widening is strictly SOFTER — it can only release a block, never create one. Measured on live corpora: +5 lessons return to the gate's view (out of 427) and +13 (out of 87).
- **Lesson type now comes from the `type` field instead of being guessed from the filename** (`lesson_files.lesson_type`). The model names files `<type value>_<slug>.md` (measured: 421 of 426 files have a prefix EXACTLY equal to the field), but nine types are in use while `lesson_prefixes` knew three. The `type` field is present in 513 of 514 files — the field is reliable, the name is not. There is deliberately NO fallback to the prefix: it would preserve the very second definition this release removes. A file without the field gets an empty type — an honest result, visible in the health pulse.
- **`user` added to the default `lesson_prefixes`.** Claude Code's official auto-memory type vocabulary is `user | feedback | project | reference`, while the library default knew three of four: a `user`-typed lesson was invisible to the gate in ANY project on defaults, by construction (exactly one such lesson in each live corpus).
- **The installer no longer splits the directories by default.** `claude-memory init` without an explicit path wrote `~/.claude/memory` — a directory nobody writes to, while Claude Code keeps memory in `~/.claude/projects/<slug>/memory`. The directory split was therefore OUT-OF-THE-BOX BEHAVIOUR, not user error: the engine read an empty folder and the gate demanded a lesson that would never appear there. The path is now discovered from the host (`claude_code_env.resolve_auto_memory_dir`): confirmed by disk → taken silently; merely derived → taken with an explicit caveat and WITHOUT creating the directory (an empty twin next to the real folder would mask the self-check's own complaint). From a git worktree the main checkout's directory is found correctly — memory is shared across the whole repository.
- **`[[wikilink]]` validation between lessons no longer skips lessons without a prefix** (`catalog_generate.find_broken_wikilinks`). Only targets starting with a lesson prefix were validated: in a live corpus EVERY link between kebab-named lessons went unchecked — had one dangled, nobody would have learned of it. A link is now recognised by the target's FORM (a compound slug, optional `.md`), not by a filename. On live corpora this surfaced 10 and 1 genuinely dangling links.
- **The task-close gate now recognises ALL nine of GitHub's closing keywords.** The default `task_close_pattern` knew `close/closes/closed` and `fix/fixes/fixed`, but not the `resolve/resolves/resolved` family — a third of GitHub's legitimate closing forms went silently unrecognised, and "did not recognise" is indistinguishable from "the task was never closed". Caught while moving the tracker from `.md` files to GitHub Issues. The same defect class is already recorded in the project's memory (a gate that recognises an event by an INCOMPLETE list of forms under-fires silently): the list of forms must mirror the SOURCE — GitHub's documentation — not the author's intuition. +3 tests, including one parametrised over all nine keywords and a control that a bare `(#5)` mention is not a closure.
- **`staleness` and `applies_to` moved onto the shared lesson set.** They globbed `*.md` with no exclusion of core/catalog/private files at all — a third definition of "lesson" living its own life. It did not misfire today (the core file has no frontmatter), but it is exactly the mine that went off under the gate.
### Added
- **Self-diagnostics for divergence from the host's settings** (`self_check.settings_issues`, SessionStart + CLI). Three complaints, from fatal to mundane: (1) Claude Code auto-memory is DISABLED (`autoMemoryEnabled: false` or env `CLAUDE_CODE_DISABLE_AUTO_MEMORY`) while the lesson gates are on → NOBODY can write lessons (the engine never creates them; it is a reader), Stop blocks after every commit and nothing can satisfy it — the only combination that produces a genuinely permanent deadlock, reachable via one line in settings.json; (2) `autoMemoryDirectory` is set explicitly and differs from `memory_dir`; (3) `memory_dir` is empty while the auto-memory directory holds lessons — the legacy of the broken installer default. Complaint (3) fires ONLY when the guess is confirmed by disk (zero own lessons AND the host has some): the slug rule is undocumented, and the engine has no right to guess out loud. Fail-open: missing file / broken JSON / no git → silence.
- **Self-check report**: `python3 -m claude_memory.self_check` now always prints the PICTURE, not just complaints — the engine's path, Claude Code's auto-memory path, whether they match, how many lessons are visible and of which types. Complaints answer "what is broken", and on a clean config there are none — leaving nothing to VERIFY a setup with, only a way to hear about a failure. Verification matters here: a directory split looks exactly like "all good, just no lessons yet".
- New module `claude_code_env` — knowledge about the host environment: the slug rule (reverse-engineered, verified on three live projects including Cyrillic paths), the main checkout from a worktree, `settings.json` scopes (local > project > user), `autoMemoryDirectory`, auto-memory kill switches. A guess is never presented as truth: it is either confirmed by disk or carries a caveat.
- Config field `precedent_files` — explicit precedent-accumulator files for auto-archiving. The contract "`lesson_prefixes[0]` is the precedent file" was positional and documented nowhere: a project with a different prefix order silently targeted the wrong file (live case: first prefix `urok`). `None` → historical behaviour, bit for bit.
- +70 tests, including the anti-resurrection one: `test_lesson_files.test_all_consumers_see_the_same_set` requires the catalog, the retriever and the gate to physically see ONE set. While it is green, the bug cannot return. Plus the end-to-end scenario "fresh commit → block → kebab lesson written → block released", which never existed before (or the bug would not have survived).
### Changed
- `lesson_prefixes` is **DEPRECATED** and no longer answers "what is a lesson" — `lesson_files.is_lesson_file` does, and the type comes from `type`. The field is KEPT (external consumers' configs must not break) in narrow roles: fallback for `size_warn_prefixes` and `precedent_files`, and link matching in `precedent_index`.
- `size_warn_prefixes: None` now means "all lessons" rather than "prefixed lessons": a lesson without a prefix used to grow without a single warning. The field remains a NARROWING knob.
- The `self_check` boundary is restated PER FUNCTION rather than per module: the right to run on every SessionStart belongs to `warnings()`/`run()` (configs + directory metadata, lesson content never), while `report()` lives only in the verbose CLI at the human's explicit request and does read content (the type breakdown exists for that). A module-level "never reads lessons" invariant would be refuted by its own `report()` — and the next person would "fix" the contradiction either by weakening the invariant into meaninglessness or by gutting the report.
- `settings_issues` on a healthy project sped up from 15.3 ms to 0.7 ms: the cheap "do I see any lessons" glob now runs BEFORE the git call (14 ms), so only an already-broken setup pays for the diagnosis. Reading settings.json costs 0.15 ms per scope and needs no optimising.
- Removed `catalog_generate._skip_basenames` — a dead function and a surviving fragment of the second definition of "lesson": working, one line from being called, plausible-looking. That is exactly how the disease returns — in a release whose whole point is "one definition", a ready-made copy of the old one has no right to live.

## [0.9.7] — 2026-07-16
### Added
- A warning about a bloated `description` at lesson-write time (`bloat.description_long`, PostToolUse `bloat-check`, threshold `description_warn_chars`, default 500 chars, 0 = off). `description` is the SUMMARY, and the edit gate and the retriever print it IN FULL: a few bloated lessons on one file become a wall of text that stops being read (which makes the bindings pointless). Truncating at display time is not a cure — it drops the very context the lesson was written for — so we fix the source, at the moment its author is still there. The threshold is ~2.5× the live corpora median (202 and 173 chars), so it does not target "detailed" lessons but ones where the body moved into the summary or two topics got merged (on the corpora: 5 of 427 and 12 of 87; a live example is a 1209-char description whose second half starts with "CSS issue #3", i.e. two different lessons in one file). Fires only on write — existing lessons do not nag. +3 tests.
- The engine now COMPLAINS OUT LOUD when `applies_to:` is set but yields no globs (`applies_to.unparsed_applies_to`) — previously any value it failed to understand silently produced emptiness, indistinguishable from "no bindings" (the third instance of this disease: cf. a `topic` from a foreign taxonomy silently falling into "No topic", and the Stop gate staying mute on a Russian closing phrase). Two channels: (1) IMMEDIATELY as the lesson is written — PostToolUse `post-record` (`applies_to.unparsed_warning`) names the file and the unparsed value while the defect's author is still there; (2) a MEMORY-WIDE summary — `staleness.scan_unparsed` → a section in `_stale_pending.md`, which the next SessionStart prints in full and untrottled until it is fixed (the shelf next to broken bindings). The complaint was deliberately kept OUT of the edit gate: a broken lesson is not bound to the file being edited and would fire on every edit of every file. `applies_to_value` distinguishes "field absent" (`None`) from "field present, value empty" (`""`) — the complaint rests on that distinction. Texts are overridable via `messages`. +7 tests (detection of every unparsed shape; both complaints; silence on valid lessons, on reads, and outside the memory dir).
### Changed
- List-valued config fields are normalized to their canonical form in `MemoryConfig.__post_init__` whenever the slip is UNAMBIGUOUS in intent: `retrieve_extensions: [".py"]` → `py`, `watched_dirs: ["./app/"]` → `app`, `lesson_prefixes: ["feedback_"]` → `feedback`, `staleness_skip_dirs: [".git/"]` → `.git`. Here the engine deliberately does NOT complain but understands — and that is not an exception to "never silently override what the human stated explicitly", it is that rule applied: the setting starts WORKING instead of quietly falling out. The complaint stays where intent is unclear (a typo in a KEY NAME → `self_check.typo_key_issues` asks), whereas `.py` cannot mean anything but `py`. Every case broke silently before and is reproduced: `.py` → `_path_re` built `\.(?:\.py)` (requiring `..py`) → the "lessons by path in the prompt" channel found nothing; `app/` → the same regex required `app//`; `feedback_` → `catalog_generate` produced `feedback__` and `stop_check` globbed `feedback__*.md` → the engine believed there were no lessons at all and the Stop gate stopped seeing what had been recorded (meanwhile `startswith` elsewhere still worked — the behavior disagreed between parts of the engine); `.git/` → the comparison is against the bare directory name from `os.walk` → the skip never fired and the walk descended into heavy directories. Normalization lives in `__post_init__` rather than `_coerce`: the invariant must hold for EVERY way of building a config (JSON / defaults / `dataclasses.replace`), otherwise the production path and the tests drift apart. Entries that collapse to nothing (`"."`, `"/"`, `"_"`) are dropped — an empty alternative in a regex would match anything. Zero changes on both live configs (they are already canonical). +4 tests.
- The `.claude/` directory is no longer skipped WHOLESALE by the edit gate — the rule became precise and moved from `hooks_cli` into `applies_to.find_lessons_for_path` (one rule for both channels): inside `.claude/`, only globs that themselves address `.claude/` match. The old blanket skip protected against something real (a broad `*.py` — `fnmatch` `*` matches slashes too — should not surface on tooling files), but it also silently cancelled EXPLICIT bindings to the real project code that lives there: one project keeps deploy guards in `.claude/hooks/`, another has a lesson that PRESCRIBES binding engine lessons to the vendored copy (`.claude/memory_engine/claude_memory/…`) — i.e. the engine was muting a project's own documented convention. In total 12 lessons across two projects never surfaced. On top of that the two halves disagreed: `memory_retrieve._path_re` NEVER skipped `.claude/`, so a lesson was found when the path was mentioned in a prompt, yet stayed silent when the same file was edited. Both truths now coexist. Verified on the live projects: it fires on exactly the explicitly bound files (8 of 72 and 6 of 33 files under `.claude/`), with 0 accidental hits from broad globs; the pre-existing regression lock (`test_pre_edit_gate_fires_in_worktree_but_skips_tooling`) stays green unchanged. +3 tests.
### Fixed
- Lesson date fields (`reverify_after`, `archived_on`) were read by a strict regex `(\d{4}-\d{2}-\d{2})` that CONFLATED "field absent" with "field present but not a date". `reverify_after: "01.01.2026"` (a natural non-ISO form) SILENTLY did not exist: the lesson looked time-bound and lived forever; an `archived_on` in the same shape never reached `archive_stale_months`. There was no complaint anywhere — not in the pulse, not in `_stale_pending`. `staleness._date_or_complaint` now yields EXACTLY three states (absent / parsed / set-to-garbage), with detector and parser being the same code (they cannot drift apart, which would give either false silence or false alarm). Shared helpers `applies_to.iso_date_or_none` + `applies_to.field_value` replace three copies of regex+try/except. Proven 1:1 on the live corpora (19 date fields: 0 old↔new diffs, 0 false complaints); +6 tests.
- Silently disabled gates and config typos are now caught by `self_check` (which already runs on EVERY SessionStart and is meant precisely to "catch settings that quietly ruin the session", yet checked a single thing): (a) `self_check.typo_key_issues` — a TYPO in a key name (`session_close_patterns` with a stray `s` → the setting is silently dropped and the English default stays; a reincarnation of the session-close gate bug that already happened). Dropping unknown keys is preserved (forward-compat: old engine × newer config), but the two populations are separated by closeness to a known name (`difflib`, stdlib): a typo is close by construction, a genuinely new key is not; keys with a leading `_` are always skipped (the established convention for notes inside JSON — a live config has `_task_close_pattern_note`, close to `task_close_pattern`; without the rule the complaint would be false every session). The message is phrased as a QUESTION ("typo, or a key from a newer version?") so a false positive asks rather than lies. (b) `self_check.bad_regex_issues` — a broken user regex (`session_close_pattern`/`task_close_pattern`): the engine caught `re.error` and returned "no match" → the gate silently turned OFF, and "off" is indistinguishable from "found nothing". (c) `self_check.bad_date_issues` — a non-ISO `model_registry_verified_on` silently disabled the lineup-check gate, even though the human FILLED IT IN, i.e. deliberately enabled it. To make this possible, `config._coerce` now REMEMBERS dropped keys in the internal `unknown_config_keys` (never accepted from JSON — otherwise a config could forge the report about its own typos): until now self_check physically could not see a typo, as the config reached it already cleaned. Zero false positives on both live configs; +7 tests.
- `applies_to` as a single STRING (`applies_to: "app/x.py"`) — a valid YAML scalar that `applies_to._applies_globs` SILENTLY dropped to `[]` since the module was born (absent from this changelog — not a regression, a hole from day one). Mechanics: a value not starting with `[` fell through to the YAML-list branch, and the very next line (usually `metadata:`) did not start with `- ` → `break` → empty list. No error, no warning — indistinguishable from "no lessons", so the lesson never surfaced on file edits. Parsing now branches on the VALUE: `[` → inline list; a non-empty scalar → one glob (quotes stripped by the shared `strip_scalar`); an EMPTY value → the YAML list below (the working format is preserved — only a non-empty value counts as a scalar). A brace value (`{a: b}`, `{app,lib}/x.py`) is not treated as a glob: neither `fnmatch` nor `glob` expands braces → it yields nothing and triggers the complaint below with the right diagnosis (otherwise it would land in the BROKEN-bindings report as "path not found — file moved?", i.e. the wrong diagnosis). Proven 1:1 on the live corpus (524 lessons with frontmatter across two projects, 288 with bindings — 0 old↔new diffs, 0 false complaints); +5 tests (scalar unquoted/quoted/with a wildcard; the "empty value + list below" boundary; the `applies_to_value` contract).

## [0.7.3] — 2026-06-29
### Fixed
- In the default `task_close_pattern`, the left boundary changed from `\b` to a negative lookbehind `(?<![\w-])`. `\b` also matches AFTER a hyphen, so a commit message like `prefixed-closes #10` (or `auto-closes #10`) was falsely read as closing task #10 — `extract_closed_task` returned `10` instead of `None`, and the close gates (`closure_reminder`, `stale_reconcile`) could show a spurious one-shot reminder. The lookbehind still accepts ordinary `Closes #id` / `Fixes #id` (at line start, after a space or a colon) but not `<word>-closes #id`. Found by a red-team check while adding a localized closure form. Affects only the generic default; consumers whose own `task_close_pattern` uses `\b` as the left boundary before the closure keyword (including anyone who copied the old default into their config) should apply the same `\b` → `(?<![\w-])` change.

## [0.7.2] — 2026-06-28
### Fixed
- `extract_closed_task` now returns the FIRST non-empty capture group of the match instead of hard-coding group 1, letting a project's `task_close_pattern` recognize task closure across DIFFERENT phrasings where the id sits on either side of the keyword. English `Closes #id` puts the id AFTER the keyword, while a localized form such as Russian `#id закрыт` puts it BEFORE — one capture group cannot cover both. The library default stays English (generic); a project supplies the form it needs via its own `task_close_pattern`. A pattern with no groups also no longer crashes (None instead of IndexError). Found on a real task close written in the Russian form `#id закрыт` (no `Closes`): both `closure_reminder` and `stale_reconcile` silently never fired. Second fix to closure detection after the `%B` fix in v0.7.1 (commit body).
