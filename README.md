# claude-memory-engine

Переиспользуемый «движок памяти уроков» для Claude Code. Ставится на любую машину одной командой и даёт проекту самообслуживаемую систему накопления и подсказки уроков. (*A reusable lessons-memory engine for Claude Code — install once, get self-maintaining lesson capture & retrieval.*)

Извлечён из инструментария проекта ЧеКи (#memory-tooling-library). В библиотеку входит ТОЛЬКО механизм; сами уроки конкретного проекта (бизнес-знания, приватные данные) в неё не входят и хранятся отдельно.

## Зачем это нужно (простыми словами)

Когда работаешь с ассистентом над проектом долго, накапливаются «уроки»: как тут принято, на какие грабли уже наступали, что нельзя ломать. Если хранить их как попало, ассистент про них забывает, а список разрастается и устаревает. Этот движок превращает уроки в аккуратную, самоподдерживающуюся систему: нужный урок сам всплывает в нужный момент, указатель собирается автоматически, старое архивируется, а параллельные сессии не затирают правки друг друга.

## Что внутри (13 механизмов и их польза)

1. **Хранилище уроков** — маленькие markdown-файлы с frontmatter (`name`, `description`, `topic`, `keywords`, `applies_to`) + одно «горячее ядро» с лимитом размера. *Польза: знания структурированы и не превращаются в свалку.*
2. **Авто-указатель CATALOG** (`catalog_generate`) — индекс собирается из поля `topic:` каждого урока, руками его не ведут. *Польза: список уроков всегда полон и не дрейфует.*
3. **Офлайн-ретривер** (`memory_retrieve`) — на каждый запрос без обращения к ИИ (ноль токенов) ранжирует уроки по словам и подсказывает релевантные, молча если ничего важного. *Польза: нужный урок всплывает сам, бесплатно.*
4. **Уроки по пути файла** (`applies_to`) — урок с глобом `applies_to` показывается перед первой правкой совпавшего файла. *Польза: предупреждение приходит ровно там, где собираешься менять код.*
5. **Авто-обслуживание** (`memory_archive`, `precedent_index`, `staleness`) — старые записи уезжают в архив, большие архивы становятся адресуемыми, а в конце сессии помечаются просроченные правила и «протухшие» привязки уроков к путям. *Польза: память не раздувается и сама подсказывает, что устарело.*
6. **Защита параллельных сессий** (`memory_concurrency`) — если две сессии правят один файл памяти, вторая получает мягкий отказ «перечитай и повтори» вместо тихой потери. *Польза: правки не пропадают.*
7. **Фиксация уроков на выходе** (`stop_check`) — мягко не даёт «закоммитил и забыл»: при завершении напоминает записать урок, если после свежего коммита заметки нет, и отдельно — если коммит закрывает задачу (`Closes #123` / `Fixes #slug`), а урока про неё не записано. Триггер и проверка настраиваемы, всё выключаемо. *Польза: знания фиксируются, пока контекст свежий, а не теряются.*
8. **Страж выбора модели суб-агентов** (`subagent_model_guard`) — не даёт запустить «рутинного» помощника на самой дорогой модели по забывчивости. *Польза: экономия без потери качества.*
9. **Замер делегирования** (`subagent_efficiency_log`) — пишет журнал запусков помощников, чтобы видеть, оправдана ли схема. *Польза: решения о делегировании — на фактах, не на догадках.*
10. **Любой язык (i18n)** (`messages`) — все операторские сообщения вынесены в каталог `messages.py` (английские дефолты) и переопределяются ключами в `messages` конфига, не трогая код. *Польза: движок работает на русском/английском/любом языке без потери скорости (словарь + `str.format`, ноль зависимостей).*
11. **Worktree-aware** — `applies_to`-страж и матчинг путей одинаково работают в обычном дереве и внутри git-worktree (`.claude/worktrees/<wt>/…`): путь релативизируется и к корню проекта, и к git-toplevel. *Польза: подсказка уроков перед правкой не «слепнет» в worktree-режиме.*
12. **Реестр моделей** (`model_registry_guard`) — на SessionStart, без токенов и сети, читает модель текущей сессии (поле события или транскрипт) и напоминает, если она вне списка известных (`known_model_substrs`) или ручная сверка линейки (`model_registry_verified_on`) старше порога. *Польза: смена/деактивация модели (как Fable) не остаётся незамеченной — конфиг и ярусы делегирования вовремя актуализируются.*
13. **Самодиагностика конфигурации** (`self_check`) — на SessionStart (каждый раз, пока не исправят) и команда `python3 -m claude_memory.self_check` сверяет, что плейсхолдеры `{x}` в каждом messages-override — подмножество дефолта. Ловит опечатки вроде `{len(cards)}` вместо `{card_count}`, которые иначе подставились бы неверно. *Польза: библиотека сама себя диагностирует — ошибка настройки видна на старте, а не «тихо ломает» функцию.* Плюс «пульс здоровья» (#2/#5) теперь проверяет и `[[wiki]]`-ссылки между уроками, и нудж «много уроков — проверь дубли» (только точные повторы) по порогу `lesson_count_warn`.

## Архитектура (три слоя)

- **Логика** — пакет `claude_memory/` (чистый Python, ноль зависимостей): движковые модули (`catalog_generate`, `memory_retrieve`, `memory_archive`, `precedent_index`, `memory_concurrency`, `session_marker_guard`, `subagent_model_guard`, `subagent_efficiency_log`, `applies_to`, `staleness`, `stop_check`) + `config.py` (вся параметризация) + `messages.py` (i18n-каталог операторских строк) + `hooks_cli.py` (единый диспетчер хук-логики) + `installer.py` (идемпотентный мердж в `settings.json`).
- **Обёртки** — `hooks/cme_hook.sh`: одна тонкая обёртка, которую Claude Code дёргает на 10 событий (`bash cme_hook.sh <событие>`); вся логика — в `hooks_cli`.
- **Данные** — уроки конкретного проекта (в движок НЕ входят, приватны).

Никаких путей конкретной машины в коде нет: всё берётся из конфига (`MemoryConfig`) и из расположения самих файлов.

## Установка (вариант A — git + install.sh)

```
git clone <repo-url> claude-memory-engine
cd claude-memory-engine
./install.sh /path/to/your/project /path/to/your/memory
```

install.sh: кладёт движок в `<project>/.claude/memory_engine/`, ставит обёртку в `<project>/.claude/hooks/`, создаёт конфиг `<project>/.claude/claude-memory.config.json` и **дописывает** регистрацию 10 хуков в `settings.json`, не затирая чужие. Повторный запуск идемпотентен (дубли не плодятся). Хуки активируются со следующей сессии Claude Code.

Аргументы по умолчанию: проект — текущий каталог, память — `~/.claude/memory`.

## Конфигурация

Все настройки — в `<project>/.claude/claude-memory.config.json`. install.sh пишет туда минимум (пути проекта); остальное берёт из дефолтов. Полный список опций со значениями по умолчанию — `examples/claude-memory.config.json`. Дефолты нейтральны (общая таксономия тем, английские сообщения); под свой проект и язык — перекройте в конфиге.

Ключевые группы опций:
- **Таксономия и файлы:** `topic_order` (слаг→заголовок раздела), `no_topic_title`, `catalog_preamble`, `catalog_auto_start`/`catalog_auto_end` (маркеры авто-блока — задайте свои, чтобы движок узнал уже существующий CATALOG), `core_file`, `catalog_file`, `session_lessons_file`, `lesson_prefixes`, `private_file_prefix`.
- **Ретривер:** `watched_dirs`, `retrieve_extensions`, `stopwords`, `retrieve_top_n`, `retrieve_threshold`, `retrieve_stem`, `retrieve_min_token`, `retrieve_body_chars`.
- **Размеры/обслуживание:** `core_size_unit` (`chars`|`bytes` — для не-латиницы честнее символы), `core_budget_bytes`, `core_warn_ratio` (ранний нудж ядра), `feedback_warn_bytes`, `size_warn_prefixes`, `size_warn_skip_archive`, `size_exempt`, `size_override`, `precedent_count_warn`, `oversize_bytes`, `precedent_archive_days`, `marker_archive_days`, `archive_dir_name`, `precedent_keyword`/`precedent_pointer`, `health_pulse_throttle`.
- **Стражи/старт/стоп:** `routine_subagent_types`, `strongest_model_substr`, `marker_limit`, `staleness_skip_dirs`, `stop_lessons_enabled`, `stop_commit_age_limit_seconds`, `task_close_lesson_gate`, `task_close_pattern`, `session_start_notes` (проектные строки-напоминания на старте сессии).
- **i18n:** `messages` — словарь `ключ → шаблон` поверх английских дефолтов из `messages.py` (например `{"unit.chars": "символов", "retrieve.hook_header": "…"}`). Плейсхолдеры в шаблонах сохраняйте.
